## 1. 요약 (Executive Summary)

개발팀은 Cursor, Claude Code 등 AI 코딩 도구를 일상적으로 사용하지만, 그 사용 기록은 각 도구에 흩어져 있습니다. 이로 인해 회사는 **매달 AI 구독료를 지출하면서도 그 돈이 실제로 어디에 쓰이는지 보여줄 데이터가 없고**, 동시에 구성원은 일일 스크럼·주간 보고를 작성할 때마다 AI 대화 기록과 Git 로그를 수동으로 되짚느라 시간을 씁니다.

본 문서는 이 두 문제를 한 번에 푸는 도구의 **6주짜리 PoC**를 제안합니다.

**이 PoC가 진짜로 검증하려는 것 (spine):**
> "개발자가 강제 없이, 자발적으로, 매일 자기 AI 사용 기록을 이 도구에 먹이는가?"

이 질문에 'Yes'가 나오면 회사는 부수 효과로 **지금까지 없던 자산** — 신뢰 가능한 AI 사용 데이터셋 — 을 6주 만에 손에 쥡니다. 이 데이터는 구독 갱신·좌석 수 결정의 근거가 되고, 향후 모든 AI 도입 의사결정의 토대가 됩니다.

**그 'Yes'를 만들어내는 미끼는 보고 시간 절감입니다.** 개발자는 "구독료 가시성"을 위해 도구를 켜지 않습니다. "스크럼 초안을 자동으로 써준다"라서 켭니다. 보고 시간 절감은 채택을 끌어내는 후크이고, 데이터셋 확보는 그 채택의 부산물입니다. PoC는 이 둘을 동시에 측정합니다.

범위는 의도적으로 좁게 잡았습니다: **Claude Code 세션 로그와 Git 커밋을 로컬에서 수집·결합하여 ① 일일 스크럼 초안(마크다운)을 자동 생성하고, ② 같은 로그에서 모델·토큰·추정 비용·세션 시간 메트릭을 함께 추출하는 것까지.** Claude Code를 첫 데이터 소스로 택한 이유는 세션 기록이 로컬에 평문 JSONL로 남아 파싱 난이도와 파손 위험이 가장 낮기 때문입니다. 결정적으로 이 JSONL에는 **모델명·토큰 사용량·타임스탬프가 이미 포함**되어 있어, 보고서용으로 읽는 그 파일에서 메트릭을 거의 추가 비용 없이 뽑을 수 있습니다(Cursor는 비공개 SQLite 구조라, 웹 기반 ChatGPT·Gemini는 로컬 로그가 없어 후속 단계로 미룹니다).

> **주의 (변함없음):** 본 PoC는 'AI 도입으로 개발 속도 X% 향상' 같은 인과적 ROI 수치를 제공하지 않습니다. 그런 수치는 통제된 비교 없이는 산출할 수 없고, 근거 없는 수치는 오히려 도구의 신뢰를 떨어뜨립니다. PoC가 약속하는 것은 ① 검증 가능한 사용 데이터셋(**모델·토큰·추정 비용 포함**) 확보 ② 보고 작성 시간 절감 ③ 자발적 채택 여부에 대한 명확한 답입니다. "세션 시간"은 **활동 시간이 아니라 세션 지속 시간(추정)**으로만 표기하며, 이를 '근무 시간'으로 포장하지 않습니다.

## 2. 문제 정의

세 문제는 우선순위가 다릅니다. 가장 비싼 것부터 적습니다.

**① (회사 비용) AI 사용 현황 데이터 부재 — 가장 비쌈**
어떤 도구를 누가 얼마나, 어떤 작업에 쓰는지에 대한 기초 데이터가 없습니다. 그 결과 구독 갱신·좌석 수·신규 도구 도입이 전부 **감(感)으로** 결정됩니다. 월 구독료가 좌석당 수만 원대로 늘어나는 상황에서, 이 의사결정을 뒷받침할 데이터가 0이라는 것이 본질적 비용입니다.

**② (구성원 비용) 보고 작성 시간 — 채택의 후크**
일일 스크럼·주간 보고 작성 시간은 현재 정확히 측정된 바 없습니다. PoC 착수 첫 주에 팀원 자가 기록으로 **베이스라인을 먼저 측정**합니다. (체감상 1인당 일 15~30분으로 추정되나, 이는 가정이며 측정으로 대체합니다.) 이 시간을 줄여주는 것이 도구를 매일 켜게 만드는 유일한 동기입니다.

**③ (장기 자산) 기록과 결과의 단절**
AI 대화(의도)와 Git 커밋(결과)이 분리되어, 나중에 '왜 이렇게 구현했는지' 맥락을 복원하기 어렵습니다. PoC는 이 둘을 시간대로 결합해 두지만, 이 가치의 본격 활용은 후속 단계입니다.

> 이 PoC는 **②(후크)로 채택을 만들어 ①(자산)을 확보**하고, ③의 데이터 토대를 부산물로 쌓습니다.

## 3. 가치 명제 — 무엇을 언제 약속하는가

ROI를 솔직하게 분리합니다. 약속하는 것과 약속하지 않는 것을 섞지 않습니다.

| 층위 | 가치 | PoC가 약속하는 것 |
|------|------|------------------|
| **단기 (측정)** | 보고 작성 시간 절감 | 6주 내 실측. 채택 후크가 실제로 작동하는지의 증거 |
| **중기 (데이터 확보)** | AI 사용 가시성 → 구독 의사결정 | 6주 내 **모델·토큰·추정 비용 데이터를 실제로 확보**(Claude Code 한정). 전사 분석·대시보드는 후속 |
| **장기 (참고)** | 의도→커밋 맥락 자산화 | PoC에서 데이터를 결합해 둠. 활용은 후속 |

**ROI에 대한 정직한 셈 (대표가 직접 할 계산을 먼저 합니다):**
보고 시간 절감만 놓고 보면 — 베이스라인 20분/일 가정 시 50% 절감은 주 ~4시간, 빌드 공수 약 120시간 → 단순 회수에 6개월 이상이 걸립니다. **보고 시간 절감 하나만으로는 이 PoC가 정당화되지 않습니다.** 이 PoC가 사는 이유는 그 부산물 두 가지입니다 — ① "6주 후 회사가 AI 구독 의사결정에 쓸, 모델별 토큰·추정 비용까지 담긴 데이터셋을 갖게 된다", ② "'개발자가 이런 도구를 자발적으로 쓰는가'라는 비싼 질문에 답을 얻는다"는 **옵션 가치**입니다. 특히 ①은 추상적 약속이 아니라 **PoC 종료 시 실제로 손에 쥐는 산출물**입니다(예: "팀이 6주간 Opus에 추정 $X, Sonnet에 $Y를 썼고, 그중 80%가 디버깅 세션이었다"). 둘 다 지금은 돈 주고도 못 사는 정보입니다.

**"데이터셋만 원하면 반나절 스크립트로 뽑으면 되지 않나?" — 맞습니다, 일회성이라면.** 토큰·비용 집계 자체는 JSONL을 긁는 짧은 스크립트로 한 번 뽑을 수 있습니다. 하지만 그건 **그 순간의 스냅샷**일 뿐이고, 누구도 매일 돌리지 않으면 한 달 뒤 죽은 데이터가 됩니다. 이 PoC가 만드는 것은 스냅샷이 아니라 **신뢰 가능하고·지속적이며·맥락(어떤 작업에 썼는지)이 붙은 데이터 흐름**입니다. 그리고 그 지속성은 정확히 "개발자가 자발적으로 매일 도구를 켠다"에서 나옵니다. 즉 채택과 데이터셋은 분리된 두 목표가 아니라 — **채택이 데이터셋을 살아있게 유지하는 엔진**입니다. 스크립트는 한 장의 사진, 이 PoC는 켜 두면 흐르는 영상입니다.

## 4. 제안: PoC 범위 (6주)

이번 PoC에서 만드는 것과 만들지 않는 것을 명확히 구분합니다.

| 구분 | 포함 (In Scope) | 제외 (Out of Scope, 후속 검토) |
|------|----------------|------------------------------|
| 데이터 수집 | Claude Code 세션 로그(JSONL) 파싱 / Git log·diff 수집 | Cursor·ChatGPT·Gemini 연동 / API 호출 이력 |
| **메트릭 추출** | **JSONL에서 모델명·토큰 사용량·세션 지속시간 추출 / 토큰×단가 → 추정 비용 산출 (Claude Code 한정)** | **멀티-AI 토큰 합산 / 실제 활동(active) 시간 정밀 측정 / 개인별 비교·랭킹** |
| 처리 | 대화-커밋 시간대 매칭 / 비밀 마스킹(검증된 비밀탐지 룰셋 기반, 직접 정규식 아님) / `.aimm-ignore` 기반 수집 제외 | PII 완전 자동 탐지 / 코드 품질 자동 평가 |
| 출력 | 일일 스크럼 초안(마크다운) / 주간 활동 요약 초안 / **초안·주간 요약에 메트릭 1줄 동봉** | 경영진용 대시보드 / ROI·MTTR 자동 산출 / Man-Month 절감 환산 |
| 대상 | 개발팀 내부 (5명 내외) | 전사 배포, 타 직군 |

### 4.1 동작 방식 — 채택은 '아무것도 안 해도 되게' 만드는 데서 갈립니다

자발적 채택의 성패는 **마찰**에서 갈립니다. 매일 CLI를 기억해서 쳐야 하는 도구는 3일이면 안 씁니다. 그래서 AIMM은 **개발자가 별도 행동을 하지 않아도 초안이 만들어지도록** 설계합니다.

- **구조:** 핵심 로직은 독립 **CLI**(수집·마스킹·메트릭·렌더). 그 위에 ① Claude Code **hook**이 자동 호출, ② `/standup` **MCP 도구**가 CLI를 감싸는 얇은 래퍼. 진입점 세 개가 같은 CLI 코어를 공유.
- **설치:** 1회 명령(`aimm init`)으로 Claude Code hook 등록. 이후 설정 불필요. (Git push 연동은 선택적 opt-in이며 레포별 git-hook 설치가 필요해 기본 경로에서 제외.)
- **트리거:** Claude Code **SessionEnd hook**으로 세션 종료 시 자동 초안 생성(기본). 사용자가 호출을 기억할 필요 없음. 보조로 `/standup` 수동 호출.
- **소비:** 다음 날 아침, 어제자 스크럼 초안이 이미 `~/aimm/draft-YYYYMMDD.md`에 준비돼 있음.
- **승인:** 사용자가 초안을 보고 → 마스킹 결과 확인 → 제출 여부·내용 직접 결정.

```
수집(Claude Code 로그, Git) → 로컬 매칭·마스킹 → 사용자 확인 → LLM 요약 → 스크럼 초안 출력
```

모든 수집과 1차 처리는 **사용자 PC 안에서만** 이루어집니다. LLM에는 마스킹을 거친 요약용 컨텍스트만 전송되며, 전송 전 사용자가 내용을 확인·승인하는 단계를 둡니다. 생성된 초안의 모든 항목에는 근거가 된 **커밋 해시**를 함께 표기하여, LLM이 실제 기록에 없는 성과를 지어내는 것을 구조적으로 억제합니다.

### 4.2 산출물 예시 (자동 생성 스크럼 초안)

> 백 마디 설명보다 이 한 장이 도구를 설명합니다. 아래는 실제 출력 형식의 mock-up입니다.

```markdown
# 일일 스크럼 — 2026-06-09 (전주성)

## 어제 한 일
- 세션 로그 JSONL 파서에 멀티라인 tool_result 처리 추가
  근거: `a3f9c21` parse: handle multiline tool_result blocks
- Git 수집기에서 머지 커밋 중복 카운트되던 버그 수정
  근거: `7b1e004` fix: skip merge commits in author stats
- (AI 대화 맥락) "JSONL 필드 구조가 버전마다 다른데 어댑터로 격리하자"는
  설계 논의 → 위 a3f9c21 구현으로 이어짐

## 오늘 할 일
- 마스킹 레이어 정규식 룰셋 1차 작성 (API 키·Bearer 토큰 패턴)
- .aimm-ignore 파서 연결

## 막힌 것 / 도움 필요
- Claude Code v1.x 로그에서 thinking 블록 포함 여부 확인 필요

## AI 사용 메트릭 (자동 추출)
- 모델: Opus 38k tok · Sonnet 12k tok  |  세션 3건, 지속(추정) 2.7h
- 추정 비용: 약 $1.40 (토큰×공시 단가 기준, 정산액 아님)

---
⚠️ 이 초안은 커밋·세션 로그 기반 자동 생성입니다. 수치·성과를 임의로 추가하지 않았으며,
   메트릭은 로그의 토큰 값에서 산출(활동 시간 아님)했고, 제출 전 본인 검토가 필요합니다.
```

> 메트릭은 **새로 수집하는 게 아니라** §4.1 파이프라인이 이미 읽는 JSONL의 `model`·`usage.*_tokens`·타임스탬프 필드에서 파생됩니다. 비용은 모델별 공시 단가 테이블을 곱한 **추정치**이며, 실제 청구액과 다를 수 있음을 항상 함께 표기합니다.

### 4.3 데이터 경계 & 신뢰 (무엇이 PC를 떠나는가)

이 도구의 신뢰는 "무엇이 외부로 나가는가"를 명확히 하는 데서 결정됩니다. CEO·구성원이 가장 먼저 묻는 질문이므로 먼저 답합니다.

- **외부 LLM으로 나가는 것 (최소화):** 커밋 메시지, diff 요약, AI 대화의 요약 텍스트 — **마스킹을 거친 뒤에만**. 소스 파일 전체·원본 비밀·전체 대화 로그는 전송하지 않습니다.
- **PC를 절대 떠나지 않는 것:** 원본 JSONL, 전체 diff, 개인 활동 원자료. 모두 본인 PC 로컬에만 존재.
- **사내 AI 사용 정책 부합:** 본 PoC는 착수 전, "코드 파생 컨텍스트를 외부 LLM에 전송하는 것"이 사내 정책 범위 내인지 확인·승인을 받습니다. (정책상 외부 전송이 불가하면, 로컬 LLM 옵션 또는 사내 API 엔드포인트로 전환 가능하도록 LLM 호출을 어댑터로 격리.)
- **마스킹은 fail-closed:** 마스커가 에러를 내거나 불확실하면 **LLM 전송을 차단**합니다. 절대 마스킹 없이 통과(fail-open)시키지 않습니다.
- **가림 가시성:** 전송 전 승인 화면에 "N개 비밀 가림"과 실제로 나가는 컨텍스트를 그대로 보여줍니다. 사용자가 보고 승인해야만 전송됩니다.

### 4.4 빈 날·실패 상태 처리 (채택을 지키는 graceful 동작)

조용한 날 깨진 초안이 한 번 나오면 채택이 죽습니다. 그래서 비-해피패스를 명시 범위에 둡니다.

- **빈 날 (커밋 0·세션 없음):** 빈/깨진 초안 대신 "오늘 기록된 활동 없음" 명시 출력.
- **부분 데이터:** 커밋만 있고 세션 없음(또는 반대)이면 있는 것만으로 초안 생성, 없는 쪽은 조용히 생략.
- **LLM 생성 실패 (타임아웃·빈 응답·거부·깨진 JSON):** 빈 초안을 내지 않고, "초안 생성 실패 — 원시 활동(커밋 목록)만 첨부" 형태로 폴백 + 가시적 에러 노트.
- **자동 트리거 실패는 조용히 죽지 않음:** 실패 시 초안 파일에 에러 노트를 남겨 "도구가 안 도는데 모르고 방치"를 방지.
- **Day boundary & 타임존:** JSONL 타임스탬프는 UTC일 수 있으므로 **로컬 타임존(KST)으로 변환 후** 일자 귀속(초안·메트릭 집계 모두). 자정을 넘는 세션은 "세션 시작 시각 기준" 하루로 귀속, 규칙을 초안 하단에 1줄 명시.
- **트리거 시점 vs 소비 시점 일자 일치:** SessionEnd는 세션 종료 시점에 돌지만, 초안은 **항상 "완결된 어제" 1개를 정본**으로 둠. 밤 세션이 끝나면 "오늘(진행 중)" 초안을 부분 갱신하되, 아침 소비용 정본은 어제자 1개로 고정해 정본이 둘이 되는 혼동을 방지.

## 5. 성공 지표 및 측정 방법

PoC의 성패는 아래 지표로만 판단합니다. **1차 지표(채택)가 합격선을 넘지 못하면, 보고 시간 절감이 아무리 좋아도 No-Go입니다.** spine이 채택이기 때문입니다.

| 우선순위 | 지표 | 측정 방법 | 목표 (6주차) |
|---------|------|----------|-------------|
| **1차 (spine)** | 자발적 채택 | 사용 로그(강제 사용 금지) + 팀원 정성 인터뷰 | 팀원 다수가 주 3회 이상 자발 사용, "없으면 아쉽다" 반응 |
| 2차 (후크) | 보고 작성 시간 절감 | 도입 전 1주 vs 도입 후 2주, 팀원 자가 기록 | 50% 이상 감소 |
| 2차 | 초안 채택률 | 수정 없이/경미한 수정으로 제출한 비율 | 70% 이상 |
| 보조 | 만족도 | 팀원 대상 5점 척도 설문 | 평균 4.0 이상 |
| 보조 (자산 검증) | 데이터셋 신뢰도 | 표본 초안의 커밋 근거 + 메트릭(토큰·모델) 추출값을 로그 원본과 수동 대조 | 근거·메트릭 오류율 5% 미만 |

> **n=5의 정직한 처리:** 표본이 5명이라 비율 지표(60%, 70%)는 통계적 의미보다 **방향성 신호**입니다. 그래서 1차 지표는 숫자 단독이 아니라 **"사용을 멈춘 사람은 왜 멈췄나"를 1:1 인터뷰로 같이 봅니다.**

> **호손 효과 차단 — '손 떼기' 자연 실험(필수):** 작성자가 만든 도구를 작성자가 지켜보는 5명이 쓰면, "주 3회 사용"과 "없으면 아쉽다"는 도구가 좋아서가 아니라 **사회적 응답**으로 나올 수 있습니다. 이를 분리하기 위해 **6주차에 작성자가 명시적으로 손을 뗍니다** — 리마인드·독려를 중단하고(또는 자동 트리거를 조용히 끄고) 며칠간 *누가 스스로 다시 켜서 쓰는가*를 관찰합니다. Go/No-Go는 이 무개입 구간의 자발 사용으로 최종 판단합니다. 만든 사람 앞에서의 "좋아요"가 아니라, 아무도 안 볼 때의 행동이 진짜 채택 신호입니다.

4주차에 중간 데모를 열어 Go/No-Go를 1차 판단하고, 6주차에 위 지표로 최종 평가합니다. 목표 미달 시 후속 단계로 확대하지 않고 종료하며, 매몰 비용은 개발 인력 6주 × 50% 공수로 제한됩니다.

## 6. 리스크와 한계

| 리스크 | 내용 | 대응 |
|--------|------|------|
| 비공식 데이터 소스 | Claude Code 로그(JSONL)는 평문이라 위험이 낮으나 공식 보장 스펙은 아니며, 업데이트로 필드 구조가 바뀔 수 있음 | 파서를 어댑터로 격리, 특정 버전 기준 개발. 파손 시 해당 소스만 비활성화되도록 설계 |
| **채택 실패 (최대 리스크)** | 후크(보고 절감)가 약하거나 마찰이 크면 자발 사용이 안 일어나고 spine이 무너짐 | **마찰 제로 설계(자동 트리거), 1~2주차에 저비용 프로토타입으로 채택 신호 먼저 확인**(§7) |
| 마스킹의 한계 | 룰셋 기반 마스킹도 완벽하지 않음. 변형된 비밀 정보는 누락될 수 있음 | 검증된 비밀탐지 룰셋(gitleaks/detect-secrets) 채용 + 사내 커스텀 룰, **fail-closed**(마스커 에러 시 전송 차단), 전송 전 사용자 확인 필수화, `.aimm-ignore`로 민감 저장소 원천 제외, 전송 표면 최소화(§4.3). '완벽한 차단'을 주장하지 않음 |
| 멀티 머신 | 회사·집 등 여러 PC로 작업하면 JSONL이 분산돼 하루 기록이 쪼개짐 | PoC는 1인 1머신 가정으로 측정. 멀티 머신 병합은 한계로 명시하고 후속 검토 |
| 사내 정책 위반 | 코드 파생 컨텍스트의 외부 LLM 전송이 사내 AI 정책에 어긋날 수 있음 | 착수 전 정책 부합 확인·승인(§4.3). 불가 시 LLM 호출 어댑터를 로컬/사내 엔드포인트로 전환 |
| 감시 도구로의 오인 | 개인 활동 기록 수집이 감시로 받아들여져 도입 자체를 무산시킬 수 있음 | 개인 데이터는 본인 PC에만 저장·본인만 열람, 제출 여부·내용은 본인 결정. 팀 단위 집계는 PoC 범위 제외. **"먼저 개인을 돕고, 회사 가시성은 나중에 옵트인"** 원칙 명시 |
| LLM의 과장 생성 | LLM이 초안을 실제보다 부풀려 작성할 위험 | 모든 항목에 커밋 근거 표기, '수치 임의 생성 금지' 프롬프트 제약, 최종 책임은 제출자 본인 |
| **시간 메트릭의 함정** | 타임스탬프로 얻는 건 세션 wall-clock 길이이지 실제 몰입 시간이 아님. '근무 시간'으로 오독되면 거짓 정밀도 | "세션 지속(추정)"으로만 라벨링, '활동 시간/근무 시간' 표현 금지. 개인 평가 용도 사용 명시적 배제 |
| **추정 비용의 정확도** | 토큰×공시 단가는 추정치이며 실제 청구액(할인·캐시·플랜)과 다를 수 있음 | 모든 비용 옆에 "추정·정산액 아님" 병기, 단가 테이블 버전 명시. 절대값보다 **상대 비교·추세** 용도로 사용 |
| 지표의 한계 | 보고 시간 절감은 측정 가능하나 'AI로 인한 생산성 향상'의 인과 입증은 PoC로 불가능 | 보고 시 상관 데이터와 인과 주장을 명확히 구분 |

## 7. 일정 및 리소스

투입 인력은 개발 1명(전주성), 업무 시간의 50%를 6주간. 별도 인프라 비용 없음(로컬 실행). LLM API 비용은 **요약 모델로 저가 모델(예: Haiku/Sonnet 급)을 사용, 1인당 일 1회 요약 기준 월 수천 원 수준, 팀 5명 합산 월 5만 원 한도** 내 집행으로 추정(요약은 Opus 불필요. 파일럿 기간 실측 후 보고). 추정 비용 산출 시 **캐시 토큰(cache_read는 정가의 ~10%)을 단가 테이블에 반영**해 과대 추정을 방지.

> **사전 기술 스파이크 (2026-06-10 실시 — 완료):**
> - **① JSONL 필드 — 확인됨.** 실제 세션 로그에서 `message.model`(예: `claude-opus-4-8`), `message.usage.{input_tokens, output_tokens, cache_read_input_tokens, cache_creation_input_tokens}`, `timestamp`(ISO 8601 **UTC**) 모두 실재 확인. → 메트릭은 추가 수집 없이 파생 가능. **캐시 토큰이 별도 필드로 존재하므로 비용 단가에 반드시 반영**(cache_read ≈ 정가 10%). timestamp가 UTC이므로 **KST 변환 후 일자 귀속**(§4.4).
> - **② SessionEnd hook — 확인됨(설정 기반).** `~/.claude/settings.json`이 `hooks` 키를 지원하며 Claude Code hook 이벤트에 SessionEnd가 포함됨. 별도 설치 없이 설정 등록만으로 자동 트리거 가능. (대안: Stop hook / 수동 `/standup`.)
> - 결론: 기획을 떠받치던 두 기술 가정이 **빌드 전 제거됨.** 외부 voice가 짚은 최대 리스크 해소.

> **"6주 빌드 전에 더 싸게 검증할 길은?"** — 진짜 리스크는 "파싱이 되는가"가 아니라 "개발자가 채택하는가"입니다. 그래서 1주차에 **완전 자동화 전, 손으로 돌리는 1회성 스크립트로 스크럼 초안 1개를 만들어 팀에 보여주고** 첫 반응을 봅니다. 반응이 싸늘하면 마찰 설계부터 다시 잡고, 좋으면 자동화에 자신 있게 투자합니다. 채택 가설을 가장 싼 단계에서 먼저 칩니다.

| 주차 | 작업 | 산출물 / 체크포인트 |
|------|------|---------------------|
| 1주 | JSONL 파서·Git 수집기 개발, 보고 시간 베이스라인 측정 시작, **수동 초안 1개로 채택 신호 확인** | 수집기 동작 확인, 베이스라인 데이터, 첫 반응 |
| 2주 | 대화-커밋 시간대 매칭 로직, **메트릭 추출(모델·토큰·세션시간·추정비용) — 같은 파서에 필드 추가** | 타임라인 결합 데이터 샘플, 첫 메트릭 추출 결과 |
| 3주 | 마스킹 레이어, `.aimm-ignore`, LLM 요약 프롬프트 1차 | 엔드투엔드 첫 자동 스크럼 초안 |
| 4주 | 초안 품질 개선, **자동 트리거(마찰 제로) 연결**, 팀 내부 데모 | 중간 Go/No-Go 판단 |
| 5주 | 팀원 5명 파일럿 시작, 피드백 반영 | 사용 로그·채택 신호 수집 개시 |
| 6주 | 파일럿 계속, 지표 집계 및 최종 평가 | PoC 결과 보고서, 확대/종료 결정 |

## 8. PoC 성공 시 후속 단계 (참고)

아래는 PoC 지표 달성을 전제로 한 방향성이며, 이번 승인 범위에 포함되지 않습니다.

- **멀티-AI 메트릭 통합** — PoC의 Claude Code 메트릭을 Cursor(비공개 SQLite, 리버스 필요)·API 직접 호출 이력까지 확장. 웹 기반 ChatGPT·Gemini는 로컬 로그가 없어 별도 경로(공식 API·관리자 콘솔 export) 조사 필요. *이것이 "진정한 멀티-AI 타임라인" 비전의 본체이며, 별도 공수 산정 후 착수*
- **AI 구독 의사결정용 도구별 사용·비용 통계** — 옵트인 기반 익명 집계. *PoC에서 확보한 모델·토큰·비용 데이터셋이 여기서 처음 현금화됨*
- **주간/월간 요약 리포트 템플릿 + 경영 대시보드** — 개인별 비교·평가가 아닌 팀 자율 보고/집계용으로 한정

## 9. 결론

이 제안의 핵심은 **작게 만들어 빨리 검증하는 것**입니다. 다만 검증의 대상은 "보고 시간이 줄어드는가"가 아니라 **"개발자가 이런 도구를 자발적으로 매일 쓰는가, 그래서 회사가 AI 사용 데이터를 자산으로 갖게 되는가"**입니다.

보고 시간 절감은 그 채택을 끌어내는 미끼이고, 측정 가능한 부수 효과입니다. 입증되지 않은 생산성 효과는 약속하지 않습니다. 6주 후, 우리는 두 가지를 손에 쥡니다 — 도구의 채택 여부에 대한 명확한 답과, 모델별 토큰·추정 비용까지 담긴, 지금은 돈 주고도 못 사는 AI 사용 데이터셋. "진정한 멀티-AI 타임라인"은 이 한 도구(Claude Code)에서 데이터가 실제로 쌓이고 개발자가 자발적으로 쓰는 것을 확인한 다음, 그 토대 위에 확장하는 것이 정직한 순서입니다. 그 데이터를 근거로 확대 여부를 결정해 주시면 됩니다.

> **승인 요청 사항:** 개발 인력 1명 × 6주 × 업무 시간 50% 투입, LLM API 비용 팀 합산 월 5만 원 한도 집행.

- **A1 마스킹:** 직접 정규식이 아니라 **검증된 비밀탐지 룰셋(gitleaks / detect-secrets)**을 마스킹 엔진으로 채용 + 소수의 사내 커스텀 룰. 마스커는 순수 함수 + **fail-closed**.
- **A2 트리거:** **Claude Code SessionEnd hook**(자동, 기본) + **`/standup` 수동**(fallback) 병행. Git push 연동은 선택적 opt-in.
- **A3 통합 형태:** **CLI 코어**가 모든 로직(수집·마스킹·메트릭·렌더)을 보유. hook과 얇은 MCP 도구는 같은 CLI를 호출하는 진입점. 로직 한 곳, 진입점 셋.
- **A4 메트릭:** 토큰·추정 비용·세션시간은 **결정적으로 로컬 계산**하고 **LLM을 우회**해 초안에 주입. LLM은 메트릭 숫자를 보거나 생성하지 않음(환각 차단).
- **설계:** JSONL은 **단일 파싱 패스**로 정규화 중간 모델(turn·model·tokens·timestamp) 생성 → 매칭·메트릭·렌더가 공유(DRY, 스트리밍 파싱으로 메모리 상한).

**빌드 테스트 커버리지 (T1 — 구현과 함께 작성, 100% 목표)**
- **마스커 적대적 스위트(★★★ 필수):** AWS·GCP·Bearer·JWT·private key·`.env` 값 픽스처가 전부 가려지는가 + 마스커 에러 시 fail-closed(전송 차단) 검증.
- **파서 충실도:** 정상 JSONL→정규화 모델 / 스키마 변경→소스 비활성화 / 레코드 깨짐→skip+로그.
- **메트릭 정확도:** 토큰 합·추정 비용(캐시 반영)이 로그 원본과 일치(결정적).
- **렌더 상태:** 빈 날 / 부분 데이터 / day boundary(타임존) / 폭주 길이 캡.
- **트리거:** SessionEnd 실패 시 초안에 가시적 에러 노트(조용한 죽음 금지).
- **LLM 클라이언트:** 타임아웃·빈·거부·깨진 JSON → 원시 활동 폴백.
