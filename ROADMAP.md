# AIMM 로드맵 — 단일 도구에서 PC 전체 LLM 분석으로

이 문서는 AIMM의 단계별 발전 방향을 기록한다. 핵심 비전은
**"한 PC에서 사용 중인 모든 LLM을, 무엇을·언제·어떤 상황에 쓰는지 분석한다"**이며,
이를 한 번에 만들지 않고 데이터 접근이 확실한 곳부터 단계적으로 확장한다.

기획·설계 배경: [AIMM_PoC_기획서_v2.md](./AIMM_PoC_기획서_v2.md) · 사용법: [README.md](./README.md)

---

## 변하지 않는 원칙 (모든 단계 공통)

이 원칙들은 단계가 올라가도 절대 양보하지 않는다. 이게 깨지면 채택이 무너진다.

1. **로컬 우선 / 프라이버시 우선** — 수집·1차 처리는 사용자 PC 안에서만. 외부 전송은
   마스킹(fail-closed) + 사용자 승인 후에만. 개인 데이터는 본인만 열람.
2. **결정적 메트릭 (LLM 우회)** — 토큰·비용·빈도는 로컬에서 결정적으로 계산. LLM은
   그 숫자를 보거나 생성하지 않는다.
3. **서술이지 평가가 아님** — "이렇게 쓴다"는 강하게 보여주되 "잘 쓴다"는 주장하지 않는다.
   토큰·빈도는 양이지 실력이 아니다. 셀프 리뷰 자료이지 성적표가 아니다.
4. **감시 아님, 상향식** — 개인이 자기 사용을 이해/어필하는 도구. 회사용 데이터는
   옵트인 익명 집계로만. 위에서 개인을 감시하는 모드는 채택하지 않는다.
5. **어댑터 격리** — 각 LLM 도구의 저장 포맷은 어댑터 뒤로 격리. 한 소스가 깨져도
   나머지는 동작.

## Phase 0 — Claude Code 단일 분석 (완료)

**상태: 기능적 완결 (2026-06-10).**

- Claude Code 세션 로그(JSONL)에서 모델·토큰·캐시·세션시간을 파싱 → 결정적 메트릭.
- 일일 스크럼 초안, 개인 사용 분석 문서(모델 믹스·비용 추세·시간대·프로젝트별).
- LLM 요약 + 마스킹 전송 경계(드라이런/--send/폴백).
- 진입점: CLI · SessionEnd hook · **MCP 서버**(Claude Code가 `standup`·`analyze` 호출).

→ "내가 Claude Code를 어떻게 쓰는지"를 본다.

## Phase 1 — MCP 기반 분석 활용 심화 (단일 LLM)

**목표:** 이미 MCP로 연결된 Claude Code를 *분석 도구로* 제대로 쓴다. 데이터 한 소스에서
얻을 수 있는 인사이트를 끝까지 짜낸다.

- 주간/월간 분석을 LLM으로 **내러티브화** — "이번 주 당신은 디버깅에 Opus를 주로 썼고,
  오후에 집중됐다" 같은 산문(마스킹 경계 통과, 근거는 결정적 수치).
- **상황(situation) 1차 신호** — 커밋 타입(feat/fix/refactor), 편집 파일 종류, 프로젝트,
  시간대로 "어떤 작업에 AI를 썼나"를 *서술적으로* 추정. (분류는 휴리스틱, 평가 아님.)
- 추세·이상치 — 비용 급증일, 평소와 다른 패턴을 짚어줌.
- 개인 분석 문서를 **공유 가능한 산출물**(PDF/마크다운)로 — 셀프 어필 용도.

**데이터 소스:** Phase 0과 동일(Claude Code JSONL). 신규 수집 없음.
**핵심 과제:** "상황" 분류 휴리스틱의 정직한 한계 표기, 내러티브의 환각 차단.

## Phase 2 — 멀티 LLM (로컬 로그가 있는 도구부터)

**목표:** Claude Code 외 다른 AI 코딩 도구로 확장. 단, **데이터 접근이 가능한 순서대로.**

각 도구는 어댑터로 추가한다(`src/parse/<tool>.ts`). 정규화 모델은 그대로 재사용.

| 소스 | 접근 방식 | 난이도 | 토큰/비용 |
|------|----------|--------|-----------|
| Cursor | 로컬 SQLite(`state.vscdb`) 리버스 | 중~상 | 저장 여부 불확실 — 조사 필요 |
| Codex CLI / Gemini CLI / aider 등 | 각자 로컬 로그(JSON/JSONL) 어댑터 | 중 | 도구가 노출하면 가능 |
| 직접 API 호출(스크립트) | 제공사 usage API / billing export | 중 | 조직 단위 집계만 |

**원칙:** 토큰/비용을 정확히 줄 수 있는 소스만 비용 분석에 포함. 없으면 "사용
빈도·시간"만 표시하고 비용은 "미상"으로 정직하게 둔다.
**핵심 과제:** Cursor SQLite 스키마 조사(별도 공수 산정), 도구별 포맷 변동 대응(어댑터 격리).
**산출물:** "나는 Cursor는 X에, Claude Code는 Y에 쓴다" 같은 **도구 간 비교** 분석.

## Phase 3 — 설치형 프로그램: PC 전체 LLM 분석

**목표:** 설치하면 그 PC에서 사용 중인 **모든 LLM**을 자동 발견·분석. "무엇을 가장 많이
쓰는가", "어떤 LLM을 어떤 상황에 가장 많이 쓰는가"를 통합해서 보여준다.

**형태:** 데스크톱 앱 또는 백그라운드 에이전트(예: Tauri/Electron, 또는 OS 서비스 +
로컬 대시보드). CLI 코어 로직은 그대로 재사용(엔진은 이미 있음).

**구성 — 세 층의 데이터를 합친다:**

1. **도구 어댑터 층** (Phase 2의 확장) — 로컬 로그가 있는 모든 코딩 LLM 도구를
   자동 발견·파싱. 토큰/비용 정밀.
2. **OS 활동 층** — 어떤 LLM 앱(Cursor·ChatGPT 데스크톱·Claude 등)이 언제·얼마나
   포그라운드였는지. 토큰은 모르지만 **사용 시간·빈도·상황**은 잡힌다.
3. **웹 LLM 층(선택)** — ChatGPT/Gemini/Claude 웹은 **로컬 로그가 없다**(하드월).
   접근 경로: 브라우저 확장(사용 캡처), 공식 데이터 export(수동·비실시간),
   제공사 usage API(조직/API 플랜). 토큰 정밀은 확장/공식 API 없이는 불가.

**"어떤 상황에 어떤 LLM"의 상황 신호:** 활성 프로젝트/레포, 편집 파일 종류, 시간대,
포그라운드 앱, 커밋 맥락. 이걸로 "리팩토링엔 A, 글쓰기엔 B" 같은 *서술적* 패턴을 만든다.

**정직한 한계 (반드시 명시):**
- 웹 LLM의 토큰/비용은 확장 또는 공식 API 없이 못 얻는다. 그 경우 "시간·빈도"까지만.
- OS 활동 층은 "AI 앱을 켜놨다"이지 "그 시간 내내 썼다"가 아니다(세션 지속의 함정과 동일).
- "가장 많이"의 기준(토큰? 시간? 빈도? 비용?)을 사용자가 고를 수 있게 — 단일 숫자로
  순위를 매기면 오해를 부른다.

**산출물:** "이번 달 당신의 AI 사용: Claude Code 45% · Cursor 30% · ChatGPT(웹) 25%
(시간 기준). 디버깅엔 Claude, 브레인스토밍엔 ChatGPT를 가장 많이." 같은 **PC 통합 리포트.**

## 단계 간 의존성

```
Phase 0 (완료) ─▶ Phase 1 (단일 심화) ─▶ Phase 2 (멀티 LLM, 로컬 우선)
                                              │
                                              ▼
                                         Phase 3 (설치형 + OS 활동층 + 웹층)
```

- Phase 1은 Phase 0 위에서 신규 수집 없이 가능 — 가장 싸고 빠른 가치.
- Phase 2는 도구별 어댑터를 하나씩. Cursor 조사가 첫 관문.
- Phase 3는 "설치형 앱 + OS 활동 캡처 + (선택)브라우저 확장"이라는 **새 아키텍처 층**이
  필요. Phase 2의 어댑터들을 그대로 흡수한다. 가장 큰 도약이자 가장 나중.

## 무엇을 먼저 하지 말 것 (안티-목표)

- 측정 못 하는 걸 약속하지 않기 — 웹 LLM 토큰을 "곧 됩니다"라고 하지 않는다.
- 단일 순위 숫자로 "잘 쓴다" 평가하지 않기 — 항상 서술 + 다축(토큰/시간/빈도/비용).
- 감시 모드로 빠지지 않기 — 회사 가시성은 끝까지 옵트인 익명 집계.
