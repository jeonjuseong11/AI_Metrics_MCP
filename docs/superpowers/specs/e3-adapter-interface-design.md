# E3 — 멀티소스 어댑터 인터페이스 (Source Adapter) — Design

Generated by /brainstorming on 2026-06-17
Branch: main | Repo: AI_Metrics_MCP | 버전: v0.4.0 (EXPANSION E3)

> AI 사용 소스(Claude Code 등)를 하나의 계약 `SourceAdapter` 뒤로 격리한다. 오케스트레이터가
> 더 이상 "Claude Code 로그가 어디 있고 어떻게 읽는지"를 직접 알지 않게 만든다. **계약 + Claude
> Code 어댑터 하나만** — Cursor(E5)·통합 뷰(E4)는 이 슬라이스에서 만들지 않는다.

## 1. 목적 & 정체성

[[aimm-poc-decisions]] CEO 플랜 시퀀싱 3번(E3). Phase 2/3(멀티 LLM·설치형)을 싸게 만드는 한 수.
지금 코어 오케스트레이터(`buildStandup`/`buildAnalysis`)는 `discoverSessionFiles()` +
`readSessionFiles()`를 직접 호출한다 — 즉 "Claude Code 파일이 어디에 있고 어떤 포맷인지"를
오케스트레이터가 알고 있다. 이 결합을 끊어, **새 소스(Cursor 등)는 인터페이스를 구현하고
주입하기만 하면** 분석 파이프라인 전체를 그대로 타게 한다.

- **사용자 체감 기능 없음**: 이건 *구조* 작업이다. CLI 출력·MCP 동작·기존 명령은 변하지 않는다.
- **가치**: E5(Cursor 어댑터)가 "인터페이스 구현 + 주입"만으로 드롭인이 된다. E4(통합 뷰)는 이
  계약을 소비할 토대를 얻는다.
- 척추(상속): 어댑터 격리(로드맵 원칙 5) · 로컬 우선 · 손상에 견고(abort 금지, warning 격리).

## 2. 확정된 결정 (brainstorming)

| 결정 | 값 |
|------|-----|
| 계약 범위 | **순수 구조 계약만.** `providesCost` 같은 능력(capability) 모델은 실제 소비자가 생기는 E5/E4에서 추가(YAGNI). |

| 하위호환 | 기존 `sessionFiles`/`projectsDir` 옵션 유지 → `CollectOptions`로 매핑. **CLI·hook·MCP 무변경.** |
| 디렉터리 | 새 `src/adapters/`. `parse/*`(순수 디코더)는 그대로 어댑터의 내부. |
| git 수집 | 어댑터 대상 **아님**(AI 사용 소스가 아니라 작업 맥락). 무변경. |

## 3. 계약 vs 파서 — 층위 구분 (명명)

로드맵 Phase 2는 어댑터를 `parse/<tool>.ts`로 적었으나, E3는 그보다 한 층 위다:

- **`parse/claudeCode.ts`** = 순수 *포맷 디코더*. `content: string → NormalizedSession`. I/O 없음.
- **`adapters/claudeCode.ts`** = *소스 어댑터*. 발견(discover) + 읽기(I/O) + 파싱 + 소스 정체성(id·이름)을
  묶어 `SourceAdapter` 계약을 만족. 파서를 내부적으로 사용.

오케스트레이터는 `SourceAdapter`에만 의존하고, 어떤 파서·파일구조도 모른다.

## 5. 계약 (`src/adapters/types.ts`)

```typescript
import type { ParseResult } from "../types.js";

/** 한 소스에서 사용 기록을 수집할 때의 옵션. 해석은 어댑터별(소스-특화). */
export interface CollectOptions {
  /** 명시 입력 위치(소스-특화: 파일 경로, DB 경로 등). 주어지면 자동 발견을 대체. */
  paths?: string[];
  /** 자동 발견 루트 오버라이드(테스트/커스텀 위치). */
  rootDir?: string;
}

/**
 * AI 사용 소스 어댑터 계약.
 *
 * 구현체는 자기 소스의 저장 포맷·위치를 캡슐화하고, 발견·읽기·파싱을 거쳐 정규화된
 * `ParseResult`(NormalizedSession[] + warnings)를 돌려준다. 레코드/파일 단위 손상은
 * warning으로 격리하고 **절대 throw로 전체를 중단하지 않는다**(로드맵 원칙 5, §4.4).
 */
export interface SourceAdapter {
  /** 안정적 기계 식별자. 예: "claude-code". */
  readonly id: string;
  /** 사람용 표시 이름. 예: "Claude Code". */
  readonly displayName: string;
  /** 발견 + 읽기 + 파싱 → 정규화 세션. */
  collect(opts?: CollectOptions): Promise<ParseResult>;
}
```

> **설계 노트(seam):** discover/read/parse를 `collect()` 하나로 묶은 것은 의도다. 세 단계는 소스별로
> *함께* 달라지고(Cursor는 DB를 발견→SQL로 읽음→행을 파싱; Claude Code의 세 함수 중 어느 것도
> 재사용 불가), 오케스트레이터는 셋을 따로 호출할 일이 없다(`NormalizedSession[]` + warnings만 필요).
> 따라서 계약에 `discover()`를 노출하지 않고 각 어댑터의 사적 구현으로 둔다. `id`/`displayName`은
> 이번 슬라이스에서 테스트(§9 #1)만 소비하지만, 어떤 어댑터든 자기 정체성은 선언해야 하므로 계약의
> 본질적 일부다(능력 모델 `providesCost`와 달리 기계 장치가 아닌 두 상수).

## 6. Claude Code 어댑터 (`src/adapters/claudeCode.ts`)

```typescript
import { discoverSessionFiles } from "../fs/discover.js";
import { readSessionFiles } from "../fs/sessions.js";
import type { ParseResult } from "../types.js";
import type { CollectOptions, SourceAdapter } from "./types.js";

/**
 * Claude Code 소스 어댑터. 기존 발견·읽기·파싱을 합성해 SourceAdapter 계약을 만족.
 * "명시 경로 > 자동 발견" 정책은 소스-특화이므로 여기(어댑터)에 둔다.
 */
export const claudeCodeAdapter: SourceAdapter = {
  id: "claude-code",
  displayName: "Claude Code",
  async collect(opts: CollectOptions = {}): Promise<ParseResult> {
    const files = opts.paths ?? (await discoverSessionFiles(opts.rootDir));
    return readSessionFiles(files);
  },
};
```

- `discoverSessionFiles(opts.rootDir)`: `rootDir`가 `undefined`면 함수의 기본 매개변수(`defaultProjectsDir()`)가
  적용된다(JS 기본값은 `undefined`에 발동). 현재 오케스트레이터의 `discoverSessionFiles(opts.projectsDir)`와 동일.
- `readSessionFiles(files)`: `projectPath` 미전달 — 현재 동작과 동일(파일 부모 디렉터리에서 프로젝트 유도).
- 시그니처: 인터페이스(§5)는 `collect(opts?: CollectOptions)`(선택 매개변수), 구현은 `collect(opts: CollectOptions = {})`
  (기본값). 기본값 있는 필수 매개변수는 선택-매개변수 계약을 구조적으로 만족하므로 strict에서 컴파일된다 — **의도된 형태**.

## 7. 오케스트레이터 수정 (`src/core/standup.ts`)

`StandupOptions`·`AnalysisBuildOptions`에 옵션 추가:

```typescript
import { claudeCodeAdapter } from "../adapters/claudeCode.js";
import type { SourceAdapter } from "../adapters/types.js";

// 두 옵션 인터페이스에 공통 추가:
  /** AI 사용 소스 어댑터. 기본 claudeCodeAdapter. 테스트·멀티소스 확장용 주입점. */
  adapter?: SourceAdapter;
```

`buildStandup`·`buildAnalysis` 양쪽에서:

```typescript
// 변경 전
const files = opts.sessionFiles ?? (await discoverSessionFiles(opts.projectsDir));
const parsed = await readSessionFiles(files);

// 변경 후
const adapter = opts.adapter ?? claudeCodeAdapter;
const collectOpts: CollectOptions = {};
if (opts.sessionFiles) collectOpts.paths = opts.sessionFiles;
if (opts.projectsDir) collectOpts.rootDir = opts.projectsDir;
const parsed = await adapter.collect(collectOpts);
```

- `discoverSessionFiles`/`readSessionFiles` import는 `standup.ts`에서 제거(어댑터로 이동). `formatParseWarning`·
  나머지 로직 불변. `standup.ts`는 `ParseResult`/`ParseWarning`(`types.js`)은 계속 import한다 — 이건
  Claude Code I/O가 아니라 **정규화된 교차-소스 계약**이므로 남는 게 맞다.
- `exactOptionalPropertyTypes`(strict, tsconfig L12 확인) 때문에 `collectOpts`는 조건부 할당으로 구성(undefined 직접 대입 금지).
- `adapter.collect()`는 다운스트림(`aggregate`/`analyze`/`formatParseWarning`)이 이미 소비하는 것과 **동일한
  `ParseResult { sessions, warnings }`**를 돌려주므로, 교체는 타입 시스템으로 end-to-end 검증된다.
- ⚠️ **빈 배열 불변식(load-bearing):** `sessionFiles: []`는 truthy이므로 `if (opts.sessionFiles)` 가드가
  `paths = []`로 매핑하고, 어댑터의 `[] ?? discover()`가 `[]`를 유지 → **자동 발견을 건너뛰고 세션 0**.
  기존 `?? ` 동작과 동일하며, 다수 기존 테스트(`sessionFiles: []` — narrative/hook)가 이 불변식을 전제한다.
  **가드를 `if (opts.sessionFiles?.length)`로 "정리"하지 말 것** — `[]`가 자동 발견으로 흘러 실 `~/.claude`를
  스캔하게 되어 테스트가 환경 의존이 된다(§9 #2가 이를 못박는다).

## 8. 에러 처리 / 폴백

- 어댑터의 `collect`는 기존 `readSessionFiles`의 보장과 동일: 개별 파일 읽기 실패는 `ParseWarning`으로 격리,
  레코드 손상은 라인 skip + warning. 절대 throw 안 함. 오케스트레이터는 `parsed.warnings`를 그대로 표면화.
- 디렉터리 부재(자동 발견 실패) → `discoverSessionFiles`가 빈 배열 → 세션 0개(기존 동작과 동일).
- 주입된 어댑터가 throw하면? 계약상 throw 금지지만, 만약을 위해 오케스트레이터 동작은 현재와 동일하게 둔다(상위
  `buildStandup`/`buildAnalysis` 호출부가 try/catch). 추가 방어는 이 슬라이스 범위 밖.

## 9. 테스트 (TDD) — `test/adapters.test.ts`

1. **계약 값**: `claudeCodeAdapter.id === "claude-code"`, `displayName === "Claude Code"`.
2. **collect({paths})**: 임시 디렉터리에 JSONL 2개(정상 1 + 손상 라인 1) 작성 →
   `collect({ paths: [f1, f2] })`가 정규화 세션을 돌려주고 손상은 warning으로 격리(throw 없음).
3. **collect() 자동 발견**: 임시 projectsDir(`<tmp>/projects/<slug>/<uuid>.jsonl`) 구성 →
   `collect({ rootDir })`가 그 파일을 발견·파싱. 같은 디렉터리에서 `collect({ rootDir, paths:[other] })`가
   `paths`로 자동 발견을 대체함도 확인.
4. **빈 paths 불변식(§7 load-bearing)**: `collect({ paths: [] })` → `{ sessions: [], warnings: [] }`,
   그리고 **자동 발견을 건너뜀**을 증명. `vi.spyOn`으로 `fs/discover`의 `discoverSessionFiles`(또는 `node:fs`의
   `readdir`)가 **호출되지 않음**을 단언(빈 배열이 디스크 스캔으로 새지 않음).
5. **없는 디렉터리(견고성)**: `collect({ rootDir: <존재하지 않는 임시경로> })` → `{ sessions: [], warnings: [] }`,
   throw 없음(어댑터 래퍼가 `discoverSessionFiles`의 빈배열 폴백을 그대로 전달).
6. **DI seam 증명(E3의 핵심) — 강화**: fake `SourceAdapter`(고유 sessionId·토큰의 손수 만든 `NormalizedSession[]`
   반환, `collect`는 `vi.fn()` spy)를 `buildAnalysis({ adapter: fake })`에 주입 →
   (a) `fake.collect` **정확히 1회 호출** 단언, (b) 결과 `UsageAnalysis.totals.sessions`가 fake가 준 세션 수와 일치,
   (c) 실 Claude Code 경로가 산출 못 할 고유값(예: 특정 sessionId/토큰)이 결과에 반영됨을 단언.
   → 데이터가 흐른 것뿐 아니라 **실제 I/O 경로가 미사용**임을 증명(단순 fake 반환만으로는 불충분 — 적대 리뷰 지적).
   `buildStandup({ adapter: fake })`도 spy 1회 호출 확인.
7. **기본 어댑터 배선(positive)**: `buildAnalysis({})`(옵션 없음)가 기본 `claudeCodeAdapter` 경로로 가는지 —
   spy 어댑터로 기본값을 가로채거나, 고정 픽스처 `sessionFiles`로 호출해 기존과 동일 `totals.sessions`를 단언.
8. **회귀 없음**: 기존 `test/narrative.test.ts`·`test/portrait.test.ts`·`test/hook.test.ts`(전부 `sessionFiles` 주입,
   `adapter` 미지정 → 기본 어댑터의 paths-매핑 경로 사용)가 그대로 통과. `cmdPortrait`도 `buildAnalysis`를 타므로
   `test/portrait.test.ts`를 회귀 게이트에 포함.

**테스트 격리 메커니즘:** 기존 코드의 주된 패턴은 **커밋된 JSONL 픽스처 + `paths`(=sessionFiles) 주입**이고,
실 임시 디렉터리를 쓰는 선례는 `test/hook.test.ts`(고정 `join(tmpdir(),"aimm-hook-test")` + `afterAll(rm)`)뿐이다.
디스크 발견이 필요한 #3/#5는 `mkdtemp(join(tmpdir(),"aimm-adapters-"))`로 격리 디렉터리를 만들고
`afterEach`에서 `rm(dir, {recursive,force})`로 정리한다(병렬·잔여 디렉터리 안전). `tsc --noEmit`(strict) 클린.

## 10. 완료 기준

- `SourceAdapter`·`CollectOptions` 계약 존재, `claudeCodeAdapter`가 구현.
- `buildStandup`/`buildAnalysis`가 어댑터 주입으로 동작하고, 기본값으로 기존 모든 호출부(CLI의 standup/analyze/**portrait**·hook·MCP) 무변경 통과.
- DI-seam 테스트가 spy로 `collect` 1회 호출 + 디스크 I/O 미사용을 증명(단순 데이터 흐름 아님).
- 빈 paths(`[]`)가 자동 발견을 건너뜀이 테스트로 못박힘.
- 전체 테스트 그린 · `tsc --noEmit` 클린 · `npm run build` 성공.

## 12. 범위 밖 (이번 슬라이스 제외)

- **능력 모델**(`providesCost`·토큰 부재 처리) — E5에서 Cursor가 실제 소비자가 될 때.
- **소스 정체성 태깅**(`NormalizedSession.source`) — 멀티소스 집계가 필요한 E4에서.
- **portrait의 도구 라벨 일반화** — `portrait.ts:116`이 "도구별 사용" 표의 유일 행으로 `| Claude Code |`를
  하드코딩한다. 어댑터에 `displayName`이 생겨도 portrait는 아직 못 쓴다(세션에 source 태그가 없음 — E4 일임).
  따라서 **E3는 이 라벨을 일반화하지 않는다.** 단, L117의 stale **렌더 출력 문구**("후속 단계(E3/E5)에서
  추가" — 초상에 찍혀 사용자에게 보임)가 E3를 지목하므로 이번에 **"E4"로 1줄 정정**(테스트로 고정)해
  약속과 현실을 맞춘다.
- **`projectPath` 의미(Claude Code 프로젝트 슬러그)** — 미래 어댑터가 채워야 할 암묵 계약. E5/E4에서 형식화.
