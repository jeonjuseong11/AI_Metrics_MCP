# 단계별 릴리스 기록 체계 (Release Tracking) — Design

Generated by /brainstorming on 2026-06-15
Branch: main | Repo: AI_Metrics_MCP

> 각 EXPANSION 항목(E1~E7)을 spec→plan→구현으로 끝내지 않고 "무엇이 어떻게 바뀌었나"의
> 증거로 닫는 기록 체계. 이 문서는 **프로세스/문서 인프라** 설계이며 코드 변경이 아니다.

## 1. 목적 & 척추

AIMM의 비전은 "AI와 어떻게 일하고 성장하는지를 보여주는 살아있는 초상"이다. 이 체계는
**도구 자신의 개발사를 그 비전의 메타 증거로** 남긴다. 각 단계가 산출물을 어떻게 바꿨고,
정직하게 검증됐으며, 만드는 데 AI를 얼마나 썼는지를 기록한다.

- **독자 = 둘 다, 단일 소스.** 내부 개발 기록으로 쌓되 그대로 외부(대표·동료) 공유 가능한
  수준으로 작성한다. 따로 관리하지 않는다.
- 변하지 않는 원칙(상속): 개인 상향식 · 로컬·마스킹 우선 · *서술*이지 *평가* 아님 ·
  경영진 감시 아님.

## 2. 파일 구조

```
프로젝트 루트/
├─ CHANGELOG.md                         ← 인덱스: 버전·E·날짜·한 줄 요약 + 상세 링크
└─ docs/
   ├─ superpowers/{specs,plans}/        (기존 — 설계·계획)
   └─ releases/
      ├─ v0.1.0-phase-1-baseline.md
      └─ 2026-06-DD-v0.2.0-e1-craft-portrait.md   (앞으로)
```

- **`CHANGELOG.md`** (루트, 인덱스): [Keep a Changelog](https://keepachangelog.com) 스타일.
  역순(최신 위). 항목당 한 줄 요약 + 상세 노트 상대경로 링크.
- **`docs/releases/<파일>.md`** (개별 상세): §4 템플릿.
- **파일명 규칙**: `YYYY-MM-DD-vX.Y.Z-<slug>.md`. `<slug>`은 해당 E 번호를 포함
  (예 `e1-craft-portrait`). 백필처럼 E가 아닌 항목은 의미 슬러그(`phase-1-baseline`).
  날짜 접두사로 시간순 정렬이 자연스럽게 유지된다.

## 3. 버전 정책

- **E당 minor bump.** 현재 출하분(Phase 0 코어 + Phase 1 ①②) = **v0.1.0**.
  E1→v0.2.0, E2→v0.3.0, … E7→v0.8.0.
- **`package.json`의 `version`을 같은 슬라이스에서 함께 올린다** (release note 작성 task에 포함).
- 이 릴리스 기록 체계 자체는 **버전을 매기지 않는다.** 기능이 아니라 인프라이므로
  CHANGELOG에 별도 버전 항목을 만들지 않고, 다음 *기능* 릴리스는 E1=v0.2.0부터 시작한다.

## 4. 릴리스 노트 템플릿 (각 E)

```markdown
# E1 — AI craft 초상  (v0.2.0 · 2026-06-DD)

## 한 줄 요약
무엇이 가능해졌나 — 사용자가 보는 변화 한 문장.

## 산출물 before/after        ← 핵심 증거 (필수)
**before** (E 적용 전 CLI 출력 샘플):
```
…
```
**after** (E 적용 후 출력 샘플):
```
…
```

## 검증                        (필수)
- 테스트: 96 → NN (신규 NN개)
- `tsc --noEmit` / `npm run build` 그린
- 적대적 검증 N건 수정 (해당 시)

## AI 사용 메타 — 도그푸딩       (필수)
`aimm analyze --start YYYY-MM-DD --end YYYY-MM-DD` 결과:
- 세션 N · 입력/출력/캐시 토큰 N · 추정 비용 N · 추정 시간 N
> ⚠️ 기간 기반 근사다. 이 E만의 귀속이 아니라 같은 기간 전체 Claude Code 사용이며,
> '시간'은 세션 지속 추정이지 근무시간이 아니다.

## 런타임 벤치                  (선택 — 해당 E만)
대개 "N/A: 핫패스 아님". E6(OS 캡처)·대용량 파싱처럼 실제로 의미 있을 때만 측정값 기재.

## 본인 메모                    (선택)
회고 한 줄.
```

- **필수**: 한 줄 요약 · before/after · 검증 · AI 사용 메타.
- **선택**: 런타임 벤치 · 본인 메모.

## 5. 워크플로우 (각 E 사이클에 편입)

```
brainstorm → spec → plan → 구현 → ★릴리스 노트 작성 → 사용자 커밋
```

- **릴리스 노트 작성을 구현 plan의 마지막 task로 넣는다.** 구현 직후 작성해야
  before/after 출력 샘플과 검증 숫자가 정확하다.
- 그 task는 세 가지를 한다: ① `docs/releases/<파일>.md` 작성 ② `CHANGELOG.md` 인덱스에
  한 줄 추가 ③ `package.json` version bump.
- **커밋·푸시는 사용자가 직접.** 에이전트는 작성까지 하고 커밋 명령을 제시한다
  (저장소 규칙: git add/commit/push는 사용자).

## 6. AI 사용 메타 = 도그푸딩

AIMM은 이미 `analyze`로 세션·토큰·추정비용을 결정적으로 계산한다(캐시 단가 반영). 해당 E를
만든 기간에 대해 `aimm analyze --start … --end …`를 돌려 그 숫자를 노트에 붙인다.
도구가 자기 개발 과정을 자기 메트릭으로 기록하는 셈이다.

- **정직성 라벨 필수**(§4 템플릿의 ⚠️ 문구). 양(토큰·세션)이지 실력이 아니며, 기간
  근사이지 E별 정밀 귀속이 아니다.
- 메트릭 코어가 이미 있으므로 추가 코드 없이 명령 한 번으로 얻는다. 별도 도구 불필요(YAGNI).

## 7. 마스킹 (fail-closed)

before/after 샘플과 AI 메타에 비밀이 섞일 수 있다. 공유 가능 단일 소스이므로:

- 출력 샘플은 **이 레포 자체 데이터**(공개 가능)로 생성하거나, 외부 데이터면 AIMM의
  `maskSecrets`를 통과시킨 결과만 붙인다.
- 의심스러우면 가린다(fail-closed). 가린 자리는 가시적으로 표기.

## 8. v0.1.0 백필 (첫 적용 산출물)

이 체계의 첫 결과물로, 이미 끝난 **Phase 0 코어 + Phase 1 ①②**를 `v0.1.0` 릴리스 노트로
소급 작성한다.

- 파일: `docs/releases/v0.1.0-phase-1-baseline.md`.
- before/after는 **약식 허용**: "git 초기 상태 → 현재"로, 핵심 기능(metrics/standup/analyze/
  내러티브/상황신호)의 대표 출력 샘플 위주. 누락 없는 정밀 diff까지 요구하지 않는다.
- 검증: 현재 테스트 96 그린 · tsc 그린.
- AI 메타: 가능하면 전체 개발 기간(2026-06-10 ~ 06-14)에 대해 `aimm analyze`로 근사.
- 동시에 `CHANGELOG.md`를 신규 생성하고 v0.1.0을 첫 항목으로 넣는다.

## 9. 범위 밖 (이번 제외)

- **`aimm release` 자동 생성 명령** — git 로그 + analyze + 테스트 결과를 묶어 노트를
  반자동 생성하는 기능. 도그푸딩의 정점이지만 이것 자체가 E급 공수다. 지금은 **수동**으로
  하고, 패턴이 안정되면 별도 E로 승격한다(YAGNI).
- 런타임 성능 벤치 하니스 — 핫패스가 생기는 E에서 그때 도입.

## 10. 완료 기준 (이 체계 도입의 done)

1. `CHANGELOG.md`가 루트에 존재하고 v0.1.0 항목 + 상세 링크를 가진다.
2. `docs/releases/v0.1.0-phase-1-baseline.md`가 §4 템플릿(백필 약식)으로 작성됨.
3. 앞으로의 모든 E 구현 plan은 마지막 task로 "릴리스 노트 + CHANGELOG + version bump"를 포함한다
   (이 규칙을 E1 plan부터 적용).
